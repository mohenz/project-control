# 프로젝트 전환 워크플로우

## 1. 전환 진입
1. 사용자 요청에서 프로젝트 별칭을 식별한다.
2. `project_registry.md`에서 프로젝트 키를 확정한다.
3. 해당 상태 파일을 연다.
4. 현재 작업 목표와 마지막 기록 시점을 확인한다.

## 2. 최소 복구 체크리스트
1. 프로젝트 경로 확인
2. 저장소 상태 또는 파일 변경 상태 확인
3. 핵심 문서 1~3개 확인
4. 실행 명령 또는 포트 확인
5. 최근 변경 또는 미해결 이슈 확인

## 3. 작업 시작 규칙
- 상태 파일에 최신 정보가 있으면 그 범위까지만 확인하고 바로 작업한다.
- 상태 파일이 오래되었거나 비어 있으면 필요한 범위만 추가 탐색한다.
- 프로젝트 전환 직후에는 다른 프로젝트의 가정을 가져오지 않는다.

## 4. 작업 중 규칙
- 새로운 중요한 사실이 나오면 상태 파일 반영 대상인지 판단한다.
- 프로젝트 고유 실행 명령과 배포 방식은 상태 파일에 유지한다.
- 문서보다 코드가 최신이면 상태 파일에 그 차이를 기록한다.

## 5. 작업 종료 체크리스트
1. 완료 작업 기록
2. 검증 결과 기록
3. 남은 작업 기록
4. 다음 시작 지점 기록
5. 배포 또는 실행 상태 기록

## 6. 빠른 전환 패턴

### 기능 개선
1. 상태 파일 확인
2. 영향 파일 확인
3. 기존 동작 확인
4. 구현
5. 검증
6. 상태 파일 갱신

### 신규 작업
1. 상태 파일 확인
2. 현재 범위와 미완료 항목 확인
3. 관련 문서 확인
4. 설계/구현
5. 검증
6. 상태 파일 갱신

### 버그 수정
1. 상태 파일 확인
2. 재현 경로 확인
3. 원인 파일 확인
4. 수정
5. 회귀 확인
6. 상태 파일 갱신

## 7. 전환 실패 조건
- 프로젝트 키를 레지스트리에서 찾지 못함
- 경로가 실제로 존재하지 않음
- 상태 파일이 없고 핵심 문서도 식별되지 않음

## 8. 전환 실패 시 조치
1. 새 프로젝트 등록 필요 여부 판단
2. 레지스트리와 상태 파일 생성
3. 그 후 작업 재시작
