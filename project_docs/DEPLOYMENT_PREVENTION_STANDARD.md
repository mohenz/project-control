# 글로벌 배포 실패 예방 표준

## 목적

- 모든 등록 프로젝트의 배포 실패와 무의미한 재시도를 줄인다.
- 배포 전·중·후 검증을 동일한 순서와 판정 기준으로 수행한다.
- Hosting, 데이터 저장소, 인증 프로젝트처럼 서로 다른 운영 대상을 혼동하지 않는다.

## 적용 원칙

- 이 문서는 `project_registry.md`에 등록된 배포 가능 프로젝트 전체에 적용한다.
- 각 프로젝트는 이 표준을 복제하지 않고 프로젝트 고유 명령·대상·불변조건만 상태 파일에 기록한다.
- 배포 자동화는 실패 시 즉시 중단해야 하며 오류를 무시하거나 다음 단계로 진행하면 안 된다.
- 사용자 승인, 테스트 단계, 데이터 변경 제한은 `project_governance_rules.md`를 우선한다.

## 1. 배포 전 사전검증

### 권한과 대상

- 사용자의 명시적 배포 지시를 확인한다.
- 레지스트리와 상태 파일에서 배포 방식, 대상 프로젝트·사이트·브랜치를 확정한다.
- CLI 실행 경로와 버전을 배포 전에 확인한다.
- 전역 CLI에 의존하지 말고 프로젝트 로컬 CLI 또는 버전이 고정된 실행기(`npx tool@version` 등)를 우선한다.

### Git 상태

- `git fetch`로 원격 상태를 갱신한다.
- 작업 트리가 깨끗한지 확인한다.
- 배포 브랜치의 `HEAD`와 upstream이 일치하는지 확인한다.
- dirty, divergent, no-upstream 상태에서는 강제 푸시·강제 병합·배포를 금지한다.

### 테스트와 빌드

- 프로젝트 상태 파일의 검증 명령과 거버넌스가 요구하는 테스트를 순서대로 실행한다.
- 테스트 러너별 구성 파일과 수집 범위를 명시한다.
- 빌드는 배포 직전에 새로 생성하고 이전 산출물을 재사용하지 않는다.
- 경고와 실패를 구분하되, 실패는 수정 전까지 배포 차단 조건으로 처리한다.

### 환경과 산출물

- 필수 환경변수는 값이 아닌 키 이름과 존재 여부만 검사한다.
- 비밀값, 토큰, API 키를 로그·문서·Git diff에 출력하지 않는다.
- 빌드 산출물에서 운영 프로젝트 ID, API 기준 URL, 데이터 저장소 식별자 등 불변조건을 확인한다.
- 잘못된 프로젝트 ID, 개발 URL, 테스트 자격증명, 금지된 도메인이 포함되면 배포를 중단한다.

## 2. 배포 실행

- 상태 파일에 기록된 표준 배포 명령 하나만 사용한다.
- 배포 범위를 명시한다. 예: Hosting만 배포할 때 DB rules·functions를 함께 배포하지 않는다.
- 최초 실패 후 같은 명령을 즉시 반복하지 않는다.
- 실패 단계와 최초 오류를 분류한 뒤 원인을 수정하고 전체 사전검증부터 다시 실행한다.
- 실행 파일 미발견 시 PATH 재시도를 반복하지 않고 로컬 CLI 또는 고정 버전 실행기로 전환한다.

## 3. 배포 후 사후검증

- 운영 URL의 HTTP 성공 응답을 확인한다.
- 캐시 무효화 쿼리를 사용해 운영 HTML의 신규 자산 해시를 확인한다.
- 운영 자산 해시가 로컬 배포 산출물과 일치하는지 확인한다.
- 운영 번들에서 프로젝트별 데이터·인증·스토리지 불변조건을 다시 검사한다.
- 핵심 공개 진입 화면과 최소 smoke 시나리오를 실행한다.
- 릴리스 또는 배포 버전, 커밋, 자산 해시, 검증 결과를 상태 파일에 기록한다.

## 4. 실패 분류와 조치

| 분류 | 대표 현상 | 우선 조치 |
|---|---|---|
| 실행 경로 | 명령을 찾지 못함 | CLI 위치·고정 버전 실행기 확인 |
| 인증·권한 | 401, 403, 권한 부족 | 로그인 상태·대상 프로젝트 권한 확인 |
| 원격 상태 | push 거절, divergent | fetch 후 브랜치 상태 확인, 강제 처리 금지 |
| 테스트 구성 | 다른 러너 파일 수집, 변환 오류 | 구성 파일·testMatch·transform 명시 |
| 환경변수 | 필수 키 누락, 잘못된 프로젝트 | 값 노출 없이 키와 불변조건 검사 |
| 빌드 산출물 | 이전 번들, 잘못된 환경 포함 | 클린 빌드 후 산출물 직접 검사 |
| 운영 캐시 | 배포 후 이전 자산 노출 | cache-busting으로 HTML·자산 해시 재확인 |
| 네트워크 | timeout, DNS, 연결 실패 | 네트워크·권한 확인 후 제한된 횟수로 재시도 |

## 5. 프로젝트 상태 파일 필수 항목

- `deploy_method`: 실제 배포 명령
- `deploy_check_command`: 배포 전 통합 검증 명령
- `deploy_post_check`: 운영 검증 명령 또는 절차
- `deploy_invariants`: 반드시 유지할 프로젝트·도메인·브랜치·산출물 조건
- `deploy_abort_condition`: 즉시 중단해야 하는 조건
- `latest_deployment`: 커밋, 릴리스 버전, 운영 자산 해시

## 표준 판정

- 사전검증 실패: 배포 금지
- 배포 명령 실패: 동일 명령 반복 금지, 원인 분석 후 사전검증부터 재시작
- 사후검증 실패: 배포 완료로 보고하지 않음
- 모든 검증 통과: 상태 파일 갱신 후 완료 보고
