# 프로젝트 총괄관리 운영 룰

## 1. 목적
- 프로젝트 전환 시 문맥 복구 시간을 줄인다.
- 프로젝트별 작업 상태를 중앙에서 관리한다.
- 신규 작업과 개선 작업이 병행되어도 현재 상태를 빠르게 복원한다.

## 2. 적용 범위
- `D:\Workspace` 내 병행 운영 프로젝트 전체
- 현재 등록 프로젝트와 placeholder는 `project_registry.md` 기준

## 3. 중앙 기준 문서
- 프로젝트 식별 기준: `project_registry.md`
- 프로젝트 전환 절차: `project_switch_workflow.md`
- 프로젝트 현재 상태: `states/*.md`
- 신규 프로젝트 등록 기준: `templates/project_state_template.md`

## 4. 절대 룰
- 프로젝트 전환 시 반드시 레지스트리부터 확인한다.
- 사용자와 약속한 프로젝트 별칭은 레지스트리 기준으로 해석한다.
- 상태 파일이 있으면 전체 재분석보다 상태 파일 우선으로 문맥을 복원한다.
- 상태 파일에 없는 핵심 사실만 추가 확인한다.
- 작업 완료 후 현재 상태, 다음 작업, 리스크를 상태 파일에 반영한다.
- 프로젝트별 실행 명령, 포트, 배포 방식은 상태 파일과 레지스트리에 동시에 기록한다.
- 새 프로젝트가 추가되면 레지스트리와 상태 파일을 먼저 만든 후 작업을 시작한다.

## 5. 프로젝트 전환 트리거
- 사용자가 프로젝트명 또는 별칭만 말한 경우
- 사용자가 "프로젝트 전환", "다른 프로젝트", "이제 n8n", "defect manage로"처럼 문맥 변경을 요청한 경우
- 현재 작업 대상과 다른 경로의 파일을 기준으로 요청한 경우

## 6. 전환 진입 규칙
- 1순위 확인: 프로젝트 키, 경로, 상태 파일
- 2순위 확인: 최근 작업 요약, 진행 중 작업, 다음 작업
- 3순위 확인: 저장소 상태, 실행 포트, 핵심 문서
- 확인이 끝나면 해당 프로젝트 기준으로만 작업을 진행한다.

## 7. 전환 종료 규칙
- 아래 항목을 상태 파일에 남긴다.
- 현재 완료한 작업
- 남은 작업
- 다음 시작 지점
- 검증 결과
- 배포 여부
- 특이사항 또는 리스크

## 8. 상태 파일 갱신 시점
- 기능 구현 완료 직후
- 큰 설계 변경 직후
- 배포 직후
- 세션 종료 직전
- 다른 프로젝트로 전환하기 직전

## 8-2. Handoff 운영 규칙
- handoff는 다음 Codex 세션 또는 다른 작업자가 즉시 이어받기 위한 실행형 인수인계다.
- `/project handoff <alias>` 요청 시 레지스트리, 상태 파일, 프로젝트 경로, Git 상태를 근거로 요약한다.
- handoff에는 현재 목표, 최근 완료, 핵심 발견, 변경 파일, 검증 결과, 다음 액션, 리스크/블로커를 포함한다.
- 블로커는 `blocked_by`, 결정 필요 사항은 `required_decision`, 금지 작업은 `do_not_do`로 명확히 쓴다.
- 비밀값, 토큰, 인증정보, 개인키는 handoff에 기록하지 않는다.
- `states/*.md`의 `Handoff` 섹션은 의미 있는 작업 종료, 중단, 또는 외부 의존성 발생 시 갱신한다.

## 8-1. `project_control` 저장소 배포 규칙
- `project_control` 저장소는 운영 규칙, 전환 방식, skill 구조, 설치 방식, command 규격처럼 중앙 운영 방식이 변경될 때만 배포한다.
- 일반 프로젝트 작업 중 `states/*.md`만 바뀐 경우에는 `project_control` 저장소를 자동 배포하지 않는다.
- 상태 파일 배포는 사용자의 명시 요청이 있거나, 중앙 운영 규칙 변경과 함께 반영이 필요할 때만 수행한다.

## 9. 신규 프로젝트 등록 절차
1. 프로젝트 키 확정
2. 사용자와 사용할 별칭 확정
3. 실제 경로 등록
4. 실행 명령과 포트 등록
5. 핵심 문서 등록
6. 상태 파일 생성
7. 전환 규칙에 편입

## 10. 약속된 프로젝트 확인 방식
- 사용자가 프로젝트 별칭을 말하면 레지스트리에서 프로젝트 키를 찾는다.
- 찾은 프로젝트 키의 상태 파일을 읽고 현재 상태를 복원한다.
- 필요한 최소 점검만 수행하고 바로 작업을 시작한다.
- 별칭이 중복되거나 불명확하면 레지스트리를 기준으로 짧게 재확인한다.

## 11. AI 에이전트 데이터베이스 보안 표준 (접근 권한 제한)
- **최소 권한 부여**: AI 에이전트 전용 계정(Role)에는 `SELECT`, `INSERT`, `UPDATE` 권한만 부여하며, `DELETE` 권한은 부여하지 않는다.
- **구조 변경 및 삭제 차단**: 에이전트 계정은 테이블의 소유자(Owner)나 Superuser가 아니어야 하며, `DROP`, `ALTER`, `TRUNCATE` 등 테이블이나 데이터 전체를 날리는 명령은 데이터베이스 엔진 레벨에서 실행 불가능하도록 제한해야 한다.
- **자동 기본 권한 적용**: 향후 새로 추가되는 테이블에 대해서도 동일한 제한이 자동 적용되도록 스키마 수준에서 기본 권한(`ALTER DEFAULT PRIVILEGES`)을 관리한다.

## 12. AI 에이전트 작업 안전 및 소통 규칙 (장애 방지 표준)

### 12.1. 데이터 및 스키마 변경 안정성 (CineTube 데이터 유실 방지 기준)
- **사용자 명시적 승인 필수 원칙**: 데이터베이스 튜닝, 테이블 스키마 변경, 데이터 이관 등 데이터 무결성과 시스템 형상에 영향을 줄 수 있는 모든 작업은 실행 전에 반드시 사용자의 구체적이고 명시적인 사전 승인을 획득해야 한다. 승인을 얻지 않은 채 독단적으로 데이터를 건드리는 모든 작업을 절대 금지한다. (백업 등은 승인 획득 이후 진행할 기술적 후속 조치에 해당한다.)
- **프로세스 락 및 충돌 해소**: 포트 바인딩 에러나 데이터베이스 커넥션 타임아웃 발생 시, 단순 재시도를 반복하지 않고 연결된 프로세스 ID(PID)를 조회하여 꼬인 프로세스를 강제 종료(Kill)한 후 가동한다.
- **사용자 제어 명령 최우선 순위**: 사용자가 "그만해", "멈춰", "정지" 등의 입력을 할 경우, 실행 중인 모든 비동기 백그라운드 태스크를 즉시 중단하고 대기 상태로 전환한다.

### 12.2. 콘솔 가이드 및 UI 설명 신뢰성 (Firebase 콘솔 UI 환각 방지 기준)
- **UI 환각 설명 절대 금지**: Firebase, AWS, Supabase 등 웹 서비스의 관리자 콘솔 UI 및 동작 단계를 설명할 때, 실제 화면 상태를 검사하지 않았거나 모호한 경우에는 임의로 메뉴명이나 버튼 위치를 추측하여 작성하지 않는다.
- **검증 정보 및 링크 제공**: UI를 설명해야 할 경우 검증된 공식 문서의 관리 콘솔 URL을 직접 제공하며, 필요한 경우 사용자의 현재 스크린샷이나 화면 설명을 먼저 요청하여 대조한 후 정확한 절차를 안내한다.

### 12.3. 데이터베이스 이관 및 형상 무결성 (Firebase 마이그레이션 기준)
- **100% 스키마 동기화 원칙**: 타 플랫폼(Supabase 등)에서 데이터베이스를 이관할 때, 사용 여부나 데이터 유무를 임의로 재단하여 일부 컬렉션만 선별 이관하지 않는다. 사용하지 않더라도 스키마 형상과 데이터 무결성을 위해 빈 테이블을 포함한 전체 구조를 동일하게 이관한다.
- **NoSQL 가시성 및 쿼리 크래시 방지**: Firestore 등 데이터가 없는 컬렉션을 콘솔에 노출하지 않거나 웹앱에서 조회 시 쿼리 에러를 유발할 수 있는 플랫폼의 경우, 초기화 스크립트를 통해 물리적 플레이스홀더 문서(예: `_init_placeholder`, `user_id: "system"`)를 삽입하여 형상을 명확히 드러내고 런타임 오류를 차단한다.

### 12.4. 빌드 호환성 및 런타임 검증 (버전 제약 방지 기준)
- **버전별 빌드 내장 함수 제약 검증**: Python 버전(예: 3.10.11)별로 타임스탬프 파싱(`fromisoformat()`) 시 마이크로초 자릿수(3자리 또는 6자리 고정) 제약이 다를 수 있으므로, 데이터를 파싱할 때는 정규식 등으로 규격을 표준화하는 예외 처리를 반드시 포함한다.
- **모듈 임포트 호환성 사전 확인**: Node.js ESM 모듈 환경 등 특정 프레임워크 스펙에서 라이브러리(예: `firebase-admin`)의 네임스페이스 호출 방식이 크래시를 유발할 수 있으므로, 최신 모듈식 임포트 규격을 따르고 실행 전에 정적 분석을 수행한다.

## 13. AI 에이전트 개발 테스트 및 배포 거버넌스 규칙 (테스트 주도 릴리즈 표준)

### 13.1. 3단계 다차원 테스트 프로세스 의무화
모든 프로젝트의 주요 변경 사항(버그 픽스, 신규 기능 등)이 발생했을 때 에이전트는 아래 3단계 테스트 절차를 순차적으로 통과해야 한다.
- **1단계: Vitest 유닛/통합 테스트**: Vite 프로젝트 특성에 최적화된 Vitest 러너를 이용하여 핵심 모듈 및 도메인 로직 유닛 테스트를 수행하고 검증한다.
- **2단계: Jest 유닛/통합 테스트**: 바벨 컴파일러 셋업과 모크(Mock) 테스트 설정을 보강하여 다중 테스트 러너 간 정합성 교차 테스트를 진행한다.
- **3단계: Playwright E2E(End-to-End) 테스트**: 브라우저 가상 환경을 띄워 사용자 실제 시나리오(인증, 업로드, 삭제 등)에 대한 실기 통신 런타임 E2E 테스트를 수행한다.

### 13.2. 테스트 단계별 필수 보고서 및 피드백 루프
- 에이전트는 각 테스트 단계(Vitest -> Jest -> Playwright)가 종료될 때마다 아래 항목이 명시된 공식 테스트 결과 보고서(`test_report_*.md` 아티팩트)를 생성하여 사용자에게 제출해야 한다.
  1. **[테스트 결과]**: 실행 통계, 성공/실패 여부, 테스트 로그
  2. **[결함 사항 (Defects)]**: 발견된 이상 동작 및 원인 분석
  3. **[결함 조치 사항]**: 이상 동작에 대한 패치 및 정정 내역
- 각 단계 완료 후 사용자의 명시적 승인을 얻어야 다음 테스트 단계로 진입할 수 있다.

### 13.3. 100% 무결성 배포 제어 및 지시 대기 원칙
- **배포 금지 원칙**: 모든 테스트 절차가 완료되고 발견된 **결함 완료율이 100%**인 상황에서만 원격 호스팅(Vercel 등) 배포를 수행할 수 있다.
- **배포 승인권**: 어떠한 테스트 프로세스가 성공적으로 완료되었다 하더라도, 사용자의 **명시적인 원격 배포 지시**가 하달되기 전까지 에이전트가 독단적으로 깃 푸시 및 원격 실서버 배포를 수행하는 행위를 **절대 금지**한다. 에이전트는 로컬 검증 완료 보고 후 사용자의 배포 결정을 대기한다.

### 13.4. 단순 정적 변경에 대한 테스트 면제 예외 조항
- **예외 대상 범위**: 비즈니스 로직, 데이터베이스 쿼리, Firebase 백엔드 모듈 등의 영향이 전혀 없는 단순 텍스트 문구 변경, UI 레이아웃 위치 수정, CSS 색상/스타일링 수정 등은 본 13조의 3단계 테스트 및 보고 절차의 대상에서 완전히 면제된다.
- **예외 처리 시 배포 제한**: 3단계 테스트 및 보고서 제출 의무는 면제되나, 이 역시 **로컬 환경에서의 실기 검증 완료 후 사용자의 승인/배포 지시가 있을 때만** 원격 저장소에 푸시하고 원격 서버에 배포하는 것을 원칙으로 삼는다.

### 13.5. 글로벌 배포 실패 예방 자동화
- 모든 등록 프로젝트의 배포는 `project_docs/DEPLOYMENT_PREVENTION_STANDARD.md`를 공통 기준으로 적용한다.
- 배포 전 원격 동기화, Git 상태, 필수 테스트, 빌드, 환경변수 존재 여부, 산출물 불변조건을 하나의 사전검증 흐름으로 확인한다.
- CLI는 전역 PATH에만 의존하지 않고 프로젝트 로컬 실행 파일 또는 버전이 고정된 실행기를 우선한다.
- 프로젝트 상태 파일에는 `deploy_method`, `deploy_check_command`, `deploy_post_check`, `deploy_invariants`, `deploy_abort_condition`을 유지한다.
- 사전검증이 실패하면 배포를 금지하고, 사후검증이 실패하면 배포 완료로 보고하지 않는다.

### 13.6. 무의미한 배포 재시도 금지
- 최초 실패 후 동일 명령을 원인 확인 없이 반복 실행하지 않는다.
- 최초 오류를 실행 경로, 인증·권한, 원격 상태, 테스트 구성, 환경변수, 빌드 산출물, 운영 캐시, 네트워크 중 하나로 분류한다.
- 원인을 수정한 뒤 부분 단계가 아니라 전체 사전검증부터 다시 시작한다.
- 강제 푸시, 강제 병합, 환경변수 우회, 테스트 생략으로 배포 실패를 회피하지 않는다.

## 14. 기능 단위 및 화면 단위 소스 분리 규칙

### 14.1. 화면 단위 물리적 분리
- 로그인, 인증, 설정, 작업 대시보드, 상세 모달, 관리자 화면처럼 사용자 목적이 다른 화면은 같은 컴포넌트 파일에 섞어 작성하지 않는다.
- 화면 전환을 담당하는 컨테이너 파일은 허용하되, 각 화면의 JSX/마크업은 독립된 화면 컴포넌트 파일로 분리한다.
- 한 파일에 로그인 화면과 실제 작업 화면을 동시에 구현하는 것을 금지한다.

### 14.2. 기능 단위 프로그램 로직 분리
- 업로드, 삭제, 인증, 검색, 필터, 페이지네이션, 데이터 동기화처럼 동작 목적이 다른 기능은 화면 JSX 내부에 장황하게 직접 구현하지 않는다.
- 화면 컴포넌트는 표시와 사용자 입력 전달을 담당하고, 데이터 변경/정합성/백엔드 호출은 서비스, 훅, 컨테이너 또는 기능 모듈로 분리한다.
- 기능 추가 시 기존 화면 파일에 임시로 로직을 누적하지 말고, 먼저 적절한 기능 단위 위치를 정한 뒤 최소 코드로 연결한다.

### 14.3. 작업 전 점검 의무
- 신규 기능 또는 화면 변경 전에 현재 파일이 화면 단위와 기능 단위 책임을 동시에 갖고 있는지 확인한다.
- 책임이 섞여 있으면 기능 구현보다 분리를 먼저 수행한다.
- 검증 보고에는 어떤 화면/기능을 어떤 파일로 분리했는지 포함한다.
